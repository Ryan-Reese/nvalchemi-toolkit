## Remote Compute (HPC)
- Topology: Mac → login node (`dlcluster`) → SLURM compute node
- SSH tunnel: `ssh -f -N -L 8890:<compute-node>:8890 dlcluster` (default port 8890)
- ProxyJump (`-J`) works for SCP and remote exec, just not for the tunnel pattern
- SLURM allocation: `./start` (2 GPUs, aarch64, 4hr timeout)
- Docker must use `--network host` (not `-p`) on shared compute nodes
- Container clones from dev fork (so `restart` picks up pushed changes)

## Beta Testing Bug Fix Workflow
1. Find runtime-impactful bug in source/docs/examples (not cosmetic)
2. Check open PRs via `WebFetch` on `https://github.com/NVIDIA/nvalchemi-toolkit/pulls` to avoid overlap
3. Confirm bug exists on latest upstream: `git show origin/main:<file>`
4. Create test `.py` file on remote (see "Remote Testing Patterns" for copy/upload commands)
5. Run in container: `ssh -J dlcluster <node> 'docker exec nvalchemi python /nvalchemi-toolkit/test_bug.py'`
6. **Before/after**: run new tests with UNFIXED code first (confirm failure), then apply fix and confirm pass
7. Run existing upstream tests with `pytest` to check for regressions
8. Clean up test files on remote
9. `scripts/deploy.sh pull-changes` to sync fix to local
10. `git worktree add fix/<name> -b fix/<name> origin/main` — apply fix in worktree, commit (1 commit ahead)
11. Run `ruff format <files> && ruff check <files>` before committing — CI lint will fail otherwise
12. `git push dev fix/<name>` then `gh pr create --repo NVIDIA/nvalchemi-toolkit --draft`
13. `git checkout main && git stash pop`

## PR Review Follow-up Workflow
1. Check PR review comments via `WebFetch` on `https://api.github.com/repos/NVIDIA/nvalchemi-toolkit/pulls/<N>/comments`
2. Switch to existing worktree: `cd fix/<name>` — then `git fetch dev fix/<name> && git reset --hard dev/fix/<name>` to sync with remote (remote may have merge-with-main commits ahead of local)
3. Make changes, upload to container, run tests
4. Commit as a NEW commit (don't amend) — reviewers can see incremental changes
5. `git push dev fix/<name>` — PR updates automatically

## Container Python
- Default python in container: `/opt/venv/bin/python` (packages from Docker build)
- Dockerfile uses `UV_PROJECT_ENVIRONMENT=/opt/venv` so build packages survive bind mount
- CUDA torch installed via `uv pip install torch --reinstall --index-url https://download.pytorch.org/whl/cu130` (after uv sync, which gets CPU torch from PyPI)
- `UV_EXTRA_INDEX_URL` does NOT work for `uv sync` torch resolution — must use explicit reinstall
- `uv.toml` cannot contain `[sources]` — only `pyproject.toml` can; also overrides `[tool.uv]` from pyproject.toml
- After Dockerfile changes: must `docker-dev.sh rebuild` (or `docker build --no-cache`) on remote
- Compute node hostname changes between SLURM allocations — check `cat /tmp/nvalchemi-deploy.env`
- Always test Dockerfile changes via remote rebuild BEFORE committing to dev fork

## Remote Testing Patterns
- All SSH/SCP commands require `dangerouslyDisableSandbox: true` (sandbox blocks DNS for internal hosts)
- Git commit/push also requires `dangerouslyDisableSandbox` (sandbox blocks git config writes)
- `git fetch origin` requires `dangerouslyDisableSandbox` (sandbox blocks SSH to github.com for NVIDIA repos)
- `$TMPDIR` differs between sandbox and non-sandbox mode — don't copy files in one mode and read in the other
- Copy test file to host /tmp/: `cat <local-file> | ssh -J dlcluster <node> "cat > /tmp/nvalchemi-toolkit/<file>"` (accessible inside container if bind-mounted)
- Run in container: `ssh -J dlcluster <node> "docker exec nvalchemi python /nvalchemi-toolkit/<script>.py"`
- Run pytest in container: `ssh -J dlcluster <node> "docker exec nvalchemi bash -c 'cd /nvalchemi-toolkit && python -m pytest <path> -v --timeout=120'"`
- Container uses live source checkout at `/nvalchemi-toolkit/` — changes via `git pull` take effect immediately
- Remote has local Dockerfile/uv.toml changes — always `git stash` before switching branches
- Clean up temp test files on remote after testing: `docker exec nvalchemi rm /nvalchemi-toolkit/<file>`
- Upload file directly to container: `cat <local-file> | ssh -J dlcluster <node> 'docker exec -i nvalchemi tee /nvalchemi-toolkit/<path> > /dev/null'`
- Upload git revision to container: `git show <ref>:<file> | ssh -J dlcluster <node> 'docker exec -i nvalchemi tee /nvalchemi-toolkit/<path> > /dev/null'`
- `git show <ref>:<file> | ssh ...` may silently fail to update container files — use `cat <local-file> | ssh ...` instead
- After replacing `.py` files in container, clear pycache: `find <dir>/__pycache__ -name "<module>*.pyc" -delete`
- Restore container files after testing: `docker exec nvalchemi bash -c 'cd /nvalchemi-toolkit && git checkout -- <file>'`

## Dynamics API Gotchas
- AtomicData for dynamics must pre-allocate `forces=zeros(n,3)`, `energies=zeros(1,1)`, `velocities=zeros(n,3)`
- NPT/NPH require `batch["stress"] = torch.zeros(num_graphs, 3, 3)` set on the Batch after `from_data_list()` — this is `batch.stress` (singular), distinct from the AtomicData field `stresses` (plural)
- `atom_categories` must be 1D `torch.long` with `AtomCategory` enum values (0–10, -1), not 2D
- `HostMemory(capacity=N)` and `GPUBuffer(capacity=N, max_atoms=M, max_edges=E)` — both require capacity arg
- `NVTNoseHoover` requires `thermostat_time`; `NPT` requires both `thermostat_time` and `barostat_time`

## Batch Class Internals
- `Batch` does NOT use Pydantic's `model_construct` — use `Batch(device=..., storage=...)` to construct directly
- `Batch._storage` is set via `object.__setattr__` in `__init__`, not a Pydantic field
- `Batch.__setattr__` routes tensors to `_storage`; `__getattr__` reads from storage groups — never mutate `batch.__dict__["field"]` directly
- Under `@torch.compile`, `_rebuild` traces `batch.positions` via `__getattr__` → storage; `__dict__` mutations are invisible to the compiled graph
- Real integrators modify positions IN-PLACE on the storage tensor (e.g. `batch.positions.add_()`) — tests must do the same

## Pydantic Internals (v2.12.5)
- `object.__setattr__` attributes (e.g. `__node_keys__`) appear in `model_dump()` — Pydantic treats them as extras
- `model_copy(deep=True)` copies `__dict__` directly (preserves `object.__setattr__` attrs), does NOT call `model_post_init`
- `DataMixin.clone()` uses `model_dump()` → `model_construct()` — `model_construct` DOES call `model_post_init`
- Always verify framework behavior empirically on the remote before writing patches
